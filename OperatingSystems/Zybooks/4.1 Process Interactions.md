# Process Competition

## Concurrency

[[Concurrency]] is the act of multiple processes (or threads) executing at the same time.

How [[concurrency]] is achieved:

* **Multiple CPUs available** — processes execute in parallel
* **Single CPU** — [[concurrency]] is achieved through time-sharing

---

## Shared Data Problem

When concurrent processes access a common data area:

* The data must be protected from simultaneous modification
* Otherwise the updated data may be left in an inconsistent state

---

## [[Critical Section]] (CS)

A [[critical section]] is:

* A segment of code that cannot be entered by a process
* While another process is executing a corresponding segment of code

---

# A Software Solution to the [[Critical Section|CS]] Problem

## Requirements for a Valid Solution

Any solution to the [[critical section]] problem must satisfy:

1. **[[Mutual Exclusion]]** — only one process can execute in the CS at a time
2. **Prevent Lockout** — a process not trying to enter the CS must not block others
3. **Prevent [[Starvation]]** — processes must not be indefinitely delayed while others repeatedly enter
4. **Prevent [[Deadlock]]** — processes must not block each other forever when trying to enter the CS

---

## Proposed Solution Overview

* Each process signals intent to enter CS:
  * `c1` and `c2` are set to 1
* A third variable `will_wait` is used to break ties

---

## Step-by-Step Behavior

### 1. Initial State

Two processes (p1 and p2) share a CS.

Initial values:
```
c1 = 0
c2 = 0
will_wait = irrelevant
```

### 2. [[Mutual Exclusion]]

When both processes try to enter:
```
c1 = 1
c2 = 1
```

* `will_wait` can only be `1` or `2`
* Memory hardware ensures writes occur sequentially

### 3. [[Race Condition]] Handling

If p1 writes first, then p2 writes, setting `will_wait = 2`:

* p1 enters CS
* p2 waits

### 4. [[Deadlock]] Prevention

* Both processes may enter the wait loop
* However, one will always proceed because `will_wait` must be either `1` or `2`

### 5. Lockout Prevention

If p2 is not trying to enter CS:
```
c2 = 0
```

* p1 can repeatedly enter CS independent of `will_wait`

### 6. Starvation Prevention

When p1 is in CS and p2 is waiting:
```
c1 = c2 = 1
will_wait = 2
```

### 7. Fairness Enforcement

If p1 wants to re-enter CS, it must set:
```
will_wait = 1
```

### 8. Ensuring No Starvation

Because `will_wait = 1`:

* p1 must wait
* p2 gets to enter CS
* p1 cannot starve p2

# Process Cooperation

## Overview

Beyond competition, processes often need to cooperate.

## Typical Scenario (Producer–Consumer)

* One process (**producer**) generates data
* Another process (**consumer**) uses that data

## Coordination Requirements

1. The producer must:
   * Notify the consumer when new data is available
2. The producer must also:
   * Wait for an acknowledgment from the consumer
   * Ensure the consumer is ready before sending the next item
