# **The Thread Concept**

## Single-Threaded Processes
An application implemented as a single process follows **one path of execution** through the program.

- Whenever execution **blocks** (waiting for a resource), the **entire process blocks**.
- However, many applications contain **independent parts** that could:
  - Run **concurrently** if they could block independently.
  - Run **in parallel** on multiple CPUs.

Creating a **separate process for each short-lived activity** is inefficient due to the overhead of creating and managing separate [[PCB||PCBs]].

## What is a Thread?
A **[[Thread]]** is an instance of executing a portion of a program **within a process**, without incurring the overhead of creating and managing separate processes.

### Key Characteristics
- **Single-threaded processes**
  - Have **one [[program counter]]**.

- **Multi-threaded processes**
  - Have a **separate [[program counter]] for each thread**.
  - Each thread:
    - Advances through code **independently**
    - Can execute **different instructions from the same program**

- When **multiple CPUs** are available:
  - Threads can execute **in parallel**.



# **Typical Use of Threads**

Threads allow different tasks within an application to run **concurrently**.

## Example: Web Browser
A browser can:
- **Draw an image**
- **Wait for data from the internet**

One thread draws the image while another thread **blocks while retrieving network data**.

## Example: Word Processor
A word processor can:
- **Run a spell checker**
- **While the user types**

Two threads can operate concurrently:
- One **waits for typed characters and displays them**
- One **runs spell checking**

## Example: Internet Server
An internet server:
- Receives **many requests from different users simultaneously**
- Implements **each request as a new thread**

This allows:
- Many requests to happen **concurrently**
- Requests to **not block each other’s progress**.



# **The Thread Control Block**

Since each thread is an **independent execution activity**, some runtime information normally stored in the [[PCB]] must be **replicated for each thread**.

A **[[Thread Control Block]] (TCB)** is a data structure that stores the **dynamic execution information needed for a thread to run independently**.

Threads remain efficient because they **replicate only minimal information**, while sharing:

- Code
- Global data
- Resources
- Open files

## Information Stored in a [[Thread Control Block|TCB]]

A [[Thread Control Block|TCB]] contains copies of:

- `CPU_state`
- `pc1`
- `sp1`
- `process_state` (stored as **thread_state**)

## Thread Execution Details

1. **Single-threaded processes**
   - Have **one [[program counter]] (pc)** and **one stack pointer (sp)** in `CPU_state`.
   - `process_state` may be:
     - running
     - ready
     - blocked

2. **Multi-threaded processes**
   - `CPU_state` is **replicated in each [[Thread Control Block|TCB]]**
   - Each thread gets its own:
     - **[[Program Counter|pc]]**
     - **sp**

3. `process_state` becomes **thread_state** in each [[Thread Control Block|TCB]]  
   - Each thread can independently be:
     - running
     - ready
     - blocked

4. Thread memory access:
   - Threads access **shared code** using their **private [[Program Counter|pc]]**
   - Threads use their **private stack** through their **private sp**

Other process-level information (like memory) remains stored in the **PCB**.



# **[[User-Level Threads]] vs [[Kernel-Level Threads]]**

Threads can be implemented in **two main ways**.

## [[User-Level Threads]] (ULTs)

Threads implemented **entirely within the application**.

A **Thread Library**:
- Provides functions to:
  - Create threads
  - Destroy threads
  - Schedule threads
  - Coordinate threads

- Maintains all **TCBs inside the user process**

Important:
- The **OS kernel is not aware** of these threads.
- The kernel treats the process as **single-threaded**.

## [[Kernel-Level Threads]] (KLTs)

Threads implemented **inside the OS kernel**.

Characteristics:
- TCBs are **not directly accessible** to the application
- The application must use **kernel services** to:
  - Create threads
  - Destroy threads
  - Manage threads



# **Combining User-Level and [[Kernel-Level Threads]]**

## Advantages of [[User-Level Threads]] (ULTs)

Compared to [[kernel-level threads]]:

- **Do not require kernel cooperation**
- **Faster to create and manage**
- Applications can create **many more ULTs than KLTs**
- **Portable across operating systems** because they are implemented in user space

## Disadvantages of ULTs

Because the kernel **cannot see ULTs**:

- If **one [[User-Level Threads|ULT]] blocks**, the **entire process blocks**
- This reduces:
  - [[Concurrency]]
  - Performance
  - Application responsiveness

- ULTs **cannot take advantage of multiple CPUs**
  - The kernel sees the process as **a single thread of execution**

## Combined Approach (Modern Operating Systems)

Many modern operating systems combine both models:

- The **kernel supports a limited number of KLTs**
- Each application can create **many ULTs**

The system then:
- **Maps ULTs onto KLTs**
- Based on:
  - Thread needs
  - Available system resources
