# **Representing Resources**
The OS maintains numerous [[Resource||resources]] that processes can request and release at runtime
Examples:
- Hardware
	- Keyboards
	- Pointing Devices
	- Printers
	- Disk and tape drives
	- Cameras
	- Speakers
- Software
	- I/O Buffers
	- Locks on database tables
	- Messages
	- Timers

Most resources are represented and handled uniformly using the same data structure and operations

Similar to a [[PCB]], a [[resource control block (RCB)]]:
- is a data structure that represents a [[resource]]
- Specific implementations and contents vary between different OSs

**Generic Structure:**

- resource_description
	- Contains the description of r's properties and capabilities.
- state
	- Shows the current availability of r
	- If r is a single unit resource then this field indicates whether r is currently free or is allocated to some process
	- If r has multiple identical units, such as a pool of identical buffers, then this field keeps track of how many units are currently free and how many are allocated
- [[Waiting List||waiting_list]]
	- When a process requests r and r is currently unavailable, the process's [[PCB]] is removed from the [[Ready List||ready list]] and added to r's [[Waiting List||waiting list]]
	- The process is moved back to the ready list when r becomes available and is allocated to the process


# **[[Resource Request||Requesting a Resource]]**
A [[resource]] is **allocated** to a process if the process has access to and is able to utilize the resource

A resource is **free** if the resource may be allocated to a requesting process

When the currently running process needs a resource:
- The process invokes a `request()` function
- The **request resource function** allocates a resource r to a process p or blocks p if r is currently allocated to another process

1. If r is currently free
	- The state of r is changed to allocated
	- A pointer to r is inserted into the list of other_resources of p
2. If r is not free
	- The calling process p is blocked
	- p's [[PCB]] is moved from the RL to the waiting_list of r
	- Since the calling process p is blocked, the [[Process Scheduler|Scheduler]] must be called to select another process to run

### **The Request Resource Function:**
```python
request(r):
    if r.state == free:
        r.state = allocated
        insert r into self.other_resources
    else:
        self.state = blocked
        move self from RL to r.waiting_list
        scheduler()
```


# **[[Resource Release||Releasing a Resource]]**
When the current running process p wants to release a resource r, the process invokes a `release()` function

The **release resource function** allocates the resource r to the next process on the r's waiting list. If the waiting list is empty, r is marked as free

1. If r's waiting_list contains no processes:
	- The state of r is changed to free
	- p continues executing
2. If the waiting list of r is not empty
	- The process q at the head of the list is allocated r
	- The state of q is changed to ready and q is moved from the waiting_list to RL
	- Since a new process q is on RL, the [[Process Scheduler|Scheduler]] must be called to decide which process (p or q) should continue to run

### **The Release Resource Function**
```python
release(r):
    remove r from self.other_resources
    if r.waiting_list == empty:
        r.state = free
    else:
        remove q from head of r.waiting_list
        insert r into q.other_resources
        q.process_state = ready
        move q from r.waiting_list to RL
        scheduler()
```


# **The Scheduler**
The **scheduler function** determines which process should run and starts the process
- Its called at the end of each of the process ad resource management functions
	- [[Process Creation||Create]]
	- [[Process Destruction||Destroy]]
	- Request
	- Release

Assuming RL is implemented as a priority list, `scheduler()` function performs:
1. Find the highest priority process q on the RL.
2. Perform a context switch from p to q if either of the following conditions is met:

	- The priority of the running process p is less than the priority of q. This condition can be true when scheduler() is called from create() or from release(). In both cases the process q could have a higher priority than p.
	- The state of the running process p is blocked. This condition is true when scheduler() is called from request() and the resource is not available. The request function changes the status of p to blocked, but the process continues executing the request and scheduler functions until the context switch is performed at the end.


### **The Scheduler Function**
```python
scheduler():
    q = highest priority process on RL
    if p.priority < q.priority or p.process_state == blocked:
        perform context switch from p to q
```
