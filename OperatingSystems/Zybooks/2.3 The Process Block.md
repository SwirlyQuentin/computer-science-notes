## **Contents of the PCB**

The [[PCB]] is the instantiation of a [[process]]
- Upon creation, the OS assigns every process a unique identifier
- This identifier could be a pointer to the PCB structure or an index into an array of PCBs

Standard Implementation of a PCB:
- CPU_state
	- The current state of the CPU
	- Consists of various hardware registers and flags
	- Save information is copied back to CPU when process resumes
- process_state
	- Stores p's current state
		- running
		- ready
		- blocked
- memory
	- Describes the area of memory assigned to p
	- Simplest case: points to contiguous area of main memory
	- In case of virtual memory: could point to a hierarchy of memory pages or segments
- scheduling_information
	- Contains info used by the [[Process Scheduler|scheduler]] to decide when p should run
	- Typically records p's CPU time, real time in system, priority, and possible deadlines
- accounting_information
	- Information necessary for accounting and billing
	- Ex: amount of CPU time or memory used
- open_files
	- Keeps track of files currently open by p
- other_resources
	- Any resources such as printers that p has requested and successfully acquired
- parent
	- Parent Process, or the process that created p
- children
	- Child process, or a process created by p

**PCB Data Structure**
- [[PCB]] are stored in structs

## **Organizing PCBs**
The OS must allocate and deallocate PCBs efficiently as processes are created and destroyed

Two ways exist to organize all PCBs:

1. Array or structs
	- PCBs are marked as free or allocated which eliminates need for dynamic memory management
	- Main drawback - alot of wasted memory space to maintain a sufficient number of PCB slots
2. Array of pointers to dynamically allocated PCBs
	- Wastes little space
	- overhead of dynamic memory management to allocate each new PCB and free memory when process terminates
	- PCBs can be reused to save on dynamic overhead
		- This is less space efficient because the total space corresponds to the max number of PCBs

## **Avoiding Linked Lists**
Linked lists require dynamic memory allocation which is costly

Approach that eliminates that overhead:

 Instead of a separate linked listanchored in the parent's PCB, the links are distributed over the child PCBs such that each points to the immediate younger sibling and immediate older sibling
 
**The original 2 fields, parent and children are replaced by 4 new fields:**
- parent: points to p's single parent
- first_child: points to p's first child
- younger_sibling: points to the sibling of p created after p
- older_sibling: points to the sibling of p created before p

This requires no dynamic memory management

### **Example**

In the linked list implementation:
- Process 0 creates process 1
- The linked list points []->0, []->1
- But 0 also points to 1 because it is a child 
- This continues as more processes get created

In the implementation without linked lists:
- Process 0 creates process 1
- Only 0 points to 1
- somewhat of a linked list that you still have to traverse, but nothing needs to be created


## **Managing PCBs**

The OS maintains PCBs on various lists:
- Waiting List
	- Associated with every resource
	- Contains all processes blocked on that resource bc that resource is not available
- Ready List
	- Contains all processes that are in the ready state
	- Includes the currently running process
	- Maintains all processes sorted by their importance
		- This is expressed as an integer value called the **priority**
	- Can be simple linked list where priority of a process is current position on list
	- Can also maintain processes in different lists sorted by priorities
