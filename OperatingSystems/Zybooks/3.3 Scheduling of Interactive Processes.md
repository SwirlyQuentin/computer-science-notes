# Interactive Processes

An [[interactive process]] communicates with the user through a dialog:

* Receives input (keyboard or pointing device)
* Produces output on the user's terminal

## Primary Goal

* Respond promptly to each user input

## Key Requirement

* Must time-share the CPU using [[Preemptive Scheduling]]
  * Ensures each process gets CPU time
  * Allows all processes to make timely progress

---

# [[RR Scheduling|Round-Robin (RR)]] Scheduling Algorithm

## [[Time Quantum]] (Q)

* A small fixed time slice (typically 10–100 ms)
* Each process can use the CPU for at most Q time units

## Core Behavior

* Uses a single queue of processes
* [[Process Priority|Priority]] is based only on queue position
  * Head = highest priority

## Execution Flow

1. Process at the head runs for Q time units
2. When Q expires:
   * Process is moved to the tail of the queue
3. Next process runs for Q time units
4. Cycle repeats

---

# Multilevel (ML) Scheduling Algorithm

## Problem with [[RR Scheduling|RR]]

* Treats all processes equally, ignoring importance

## Solution

* Introduce external priorities

## Core Idea

* Maintain multiple queues, one per priority level

## Behavior

* Each priority level has its own queue
* Within each level:
  * Processes are scheduled using [[RR Scheduling|Round-Robin (RR)]]

---

# [[MLF Scheduling|Multilevel Feedback]] (MLF) Scheduling Algorithm

## Improvement Over ML

Addresses:

* [[Starvation]]
* Fairness between processes

## Key Features

* Different [[time quantum]] per priority level
* Dynamic priority adjustment

## Execution Flow

1. New process enters highest priority queue (N)
2. Runs for Q time units
3. If it exceeds Q:
   * Moves to next lower level (N−1)
4. At lower level:
   * Runs for 2Q time units
5. This continues — [[time quantum]] doubles each level down

## General Rule

* At priority level L:
```
Max time = 2^(N-L) * Q
```

## Behavior Insight

* Short processes → stay at high priority → finish quickly
* Long processes → move down → get longer but less frequent CPU time

# Performance of Interactive Scheduling Algorithms

## Response Time

* Time from user request (e.g., key press, click) → to when the response begins

## Goal

* Ensure adequate (fast) [[response time]]

## Single Process Scenario

* [[Response time]] depends on request type:
  * Typing a character → should be instant (imperceptible)
  * Heavy tasks (spell check, rendering) → may take seconds

## Multiple Processes (Time-Sharing)

[[Response time]] increases with:

* Number of processes
* Length of the [[time quantum]] (Q)
* Context switch overhead
