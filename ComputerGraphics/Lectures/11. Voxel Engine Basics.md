# **The Block Class**

A **[[Block (Voxel Block Class)|block]]** is a [[voxel]] and is represented as a **3D cube**.

---

## Core Concept

* Blocks make up the **bulk of a voxel engine**.
* They act as the **base data class** that the engine will contain.
* When [[voxel]] data for the world is loaded, **large numbers of blocks are created**.

If the game is saved:

* The engine must **store information for each block in the world**.

---

## Block Data Requirements

Each [[Block (Voxel Block Class)|block]] is:

* **Unique**
* **Dynamically alterable**

Because of this, **each [[Block (Voxel Block Class)|block]] must store its own data**.

However:

* Block data variables must be **minimized** to prevent **overhead and memory bloat**, since voxel worlds contain **millions of blocks**.

---

## Data That Does NOT Need to Be Stored

### Position

Block **position does not need to be stored**.

Instead:

* Position can be **inferred from the block's index** inside the storage array.

### Normal Vectors

Normal vectors for block faces **do not need to be stored** because:

* They can be **calculated procedurally** for each voxel side.

### Texture or Color Data

Texture coordinates and colors **do not need to be stored per block**.

Instead:

* **[[Block type]]** determines which **texture and color** should be used.

---

## Important Data to Store

### [[Active Block Flag|Active Flag]]

Each block needs to store whether it is **active or inactive**.

Purpose:

* Determines whether the block is **on or off**.
* During rendering, **inactive blocks are not rendered**.

This allows:

* **Dynamic manipulation of the voxel world** by toggling blocks.

### Block Type

Blocks also store a **type identifier**.

Purpose:

* Allows multiple block varieties (stone, grass, etc.).
* Maps the block to its **associated textures and colors**.

---

## Saving Block Data

A file system can be used to store:

* The **[[Active Block Flag|active flag]]**
* The **[[block type]]**

This data can then be:

* **Written to disk**
* **Read back into the engine**

---

## Important Design Considerations

When designing voxel storage:

* Store **as little information as possible** per block.
* Worlds may contain **millions of voxels**.
* **Procedurally generate** any data that can be derived instead of storing it.

---

## Comparison to Traditional 3D Models

In traditional 3D models:

* Most data is **stored in files and loaded at runtime**.

In voxel engines:

* The goal is to **store minimal information**.
* The engine should **reconstruct the world from small datasets** every time it loads.

---

# **Chunks**

A **[[chunk]]** is a structure used to **organize and group voxels together**.

---

## Purpose of Chunks

Voxel engines often aim to support **infinite worlds** along the X, Y, and Z axes.

However, rendering every [[voxel]] individually creates a major performance problem.

Example:

* A world of **256 × 256 × 256 voxels**
* This would require rendering **16,777,216 blocks**

This is extremely inefficient.

A good voxel engine should **minimize the number of draw calls sent to the GPU**.

---

## Chunk-Based Rendering

Chunks solve this problem.

A **[[chunk]]** bundles many blocks together so the engine can:

* **Render the entire chunk in a single [[draw call]]**

This greatly **reduces rendering overhead**.

### Typical Chunk Size

A common starting size is **16 × 16 × 16 blocks**.

| Rendering Strategy | Number of Draw Calls |
| ------------------ | -------------------- |
| Individual Blocks  | 16,777,216           |
| Chunks             | 4,096                |

---

## Chunk Trade-Offs

### [[Chunk Rebuilding]]

When a [[Block (Voxel Block Class)|block]] is modified:

* The **entire [[chunk]] must be rebuilt**
* The chunk must then be **re-rendered**

### Border Cases

If the modified block lies on a **chunk boundary**:

* **Two chunks must be rebuilt and rendered**

### Chunk Size Trade-Off

| Larger Chunks         | Smaller Chunks       |
| --------------------- | -------------------- |
| Fewer rendering calls | More rendering calls |
| Slower rebuild times  | Faster rebuild times |

Chunk size must balance **rendering performance** and **update performance**.

---

# **Advanced Rendering**

Voxel engines often use **GPU optimization techniques**.

Two common approaches are:

* **Vertex Buffers**
* **[[Display List|Display Lists]]**

Purpose:

* Avoid repeatedly sending the same rendering commands.
* Reduce **CPU → GPU traffic**.

---

# **Render Modes**

OpenGL provides **four main rendering modes**.

---

## Immediate Mode

* Every time a set of vertices is completed, a **[[draw call]] is sent to the GPU**.
* Slow performance, easy to edit dynamically.
* **Most flexible, worst performance.**

---

## [[Vertex Array|Vertex Arrays]]

* Vertices are **stored in an array** then **sent to the GPU in a single call**.
* Allows **dynamic modification** — vertices can be edited or removed.
* Faster than [[Immediate Mode]], slower than [[Display List|Display Lists]].

---

## Display Lists

* Vertices are **stored directly on the GPU**.
* Very fast, but data **cannot be dynamically modified**.
* Reduces CPU → GPU communication.
* Not suitable for frequently changing geometry.

---

## Vertex Buffer Objects (VBOs)

* Similar to [[Vertex Array|Vertex Arrays]] but faster.
* Data stored in **GPU memory**.
* Allows **dynamic modification** with more efficient rendering.
* **Harder to set up and debug.**

---

# **Rendering Chunks with Vertex Buffers**

When using **Vertex Buffer Objects**, two useful OpenGL calls are:

* `glDrawElements()`
* `glDrawArrays()`

These functions allow the engine to **group [[voxel]] data for a [[chunk]]** and **render it in a single GPU call**.

---

## Chunk Rendering Workflow

To render a chunk in **one render call**:

1. **Generate the vertex buffer** for the chunk.
2. During [[chunk]] setup or loading, create a **[[Vertex Buffer Object (VBO)|vertex buffer object]] in the renderer**.
3. Loop through the chunk's blocks.
4. For each [[Block (Voxel Block Class)|block]]:
   * If the block is **active**, add **quad data** for its faces to the vertex buffer.

Result:

* The vertex buffer contains **only active blocks**.

---

## Final Rendering Step

When the renderer draws the chunk's vertex buffer:

* **Only active blocks in the [[chunk]] are rendered**

This keeps rendering efficient while still allowing the [[voxel]] world to be **dynamic**.
