## Command Pattern

**GOF Definition:**  
Encapsulate a request as an object, thereby letting users parameterize clients with different requests, queue or log requests, and support undoable operations.

**Book Definition:**  
A **command** is a **reified method call**.

- **Reify** basically means *to make real*.
- Taking some concept and turning it into a piece of data.
- An object that you can do something with:
  - store in a variable
  - pass to a function
  - queue or log
- Similar to a **callback**.


---

# Configuring Input – Example 1

Somewhere in every game is a chunk of code that reads raw user input.

### Basic Implementation
```cpp
void InputHandler::handleInput()
{
  if (isPressed(BUTTON_X)) jump();
  else if (isPressed(BUTTON_Y)) fireGun();
  else if (isPressed(BUTTON_A)) swapWeapon();
  else if (isPressed(BUTTON_B)) lurchIneffectively();
}
```

This function gets called **once per frame** by the game loop.

This works **if we hard-wire user inputs to game actions**, but:

* Most games allow users to **configure button mappings**.

Direct calls like `jump()` and `fireGun()` need to become **something we can swap out**.

---

## Enter the Command Pattern

### Base Class
```cpp
class Command
{
public:
  virtual ~Command() {}
  virtual void execute() = 0;
};
```

This allows us to create subclasses for each different game action.

### Example Commands
```cpp
class JumpCommand : public Command
{
public:
  virtual void execute() { jump(); }
};

class FireCommand : public Command
{
public:
  virtual void execute() { fireGun(); }
};

// You get the idea...
```

---

### Input Handler with Commands

In our input handler we store **pointers to commands for each button**.
```cpp
class InputHandler
{
public:
  void handleInput();

  // Methods to bind commands...

private:
  Command* buttonX_;
  Command* buttonY_;
  Command* buttonA_;
  Command* buttonB_;
};
```

Now input handling **delegates to commands**.
```cpp
void InputHandler::handleInput()
{
  if (isPressed(BUTTON_X)) buttonX_->execute();
  else if (isPressed(BUTTON_Y)) buttonY_->execute();
  else if (isPressed(BUTTON_A)) buttonA_->execute();
  else if (isPressed(BUTTON_B)) buttonB_->execute();
}
```

### Key Idea

There is now a **layer of indirection**.

> This is the Command Pattern in a nutshell.

# Directions for Actors

### Limitation From Previous Example

The earlier example assumes:

* `jump()` and other functions are **top-level functions**
* These implicitly know how to find the **player's avatar**

This creates **tight coupling**.

Limitations:

* The *only* thing the jump command can make jump is the **player**.

---

## Fixing the Coupling

Instead of calling functions that find the object themselves, we **pass the object into the command**.

### Updated Command Interface
```cpp
class Command
{
public:
  virtual ~Command() {}
  virtual void execute(GameActor& actor) = 0;
};
```

`GameActor` represents a **character in the game**.

---

### Updated Command Implementation
```cpp
class JumpCommand : public Command
{
public:
  virtual void execute(GameActor& actor)
  {
    actor.jump();
  }
};
```

Now the **same command class can be used for any character**.

---

### Updating the Input Handler

We modify `handleInput()` so it **returns a command**.
```cpp
Command* InputHandler::handleInput()
{
  if (isPressed(BUTTON_X)) return buttonX_;
  if (isPressed(BUTTON_Y)) return buttonY_;
  if (isPressed(BUTTON_A)) return buttonA_;
  if (isPressed(BUTTON_B)) return buttonB_;

  // Nothing pressed
  return NULL;
}
```

Then another piece of code executes the command on the player.
```cpp
Command* command = inputHandler.handleInput();
if (command)
{
  command->execute(actor);
}
```

---

### Benefits of This Indirection

This design allows:

* The player to **control any actor in the scene**

More importantly:

* **AI systems can also use commands**

  * AI simply emits `Command` objects
  * Actors execute them
* Allows:

  * Different AI modules for different actors
  * Mix-and-match behaviors
  * Player **auto-pilot systems**

# Undo and Redo – Example 2

This is **the most well-known use of the Command Pattern**.

### Scenario

We are making a **single-player turn-based game** and want users to **undo moves**.

We assume we are already using commands for **input handling**.

---

## Moving a Unit
```cpp
class MoveUnitCommand : public Command
{
public:
  MoveUnitCommand(Unit* unit, int x, int y)
  : unit_(unit),
    x_(x),
    y_(y)
  {}

  virtual void execute()
  {
    unit_->moveTo(x_, y_);
  }

private:
  Unit* unit_;
  int x_, y_;
};
```

This differs from the previous implementation.

### Previous Example

We wanted to:

* **Abstract the command from the actor**
* Represent a **general action**

### Here

We want to:

* **Bind the command to a specific unit**
* Represent **a specific action at a specific moment**

Input handling code **creates a new command each time the player acts**.

> This command represents a specific move in the game's sequence of turns.

---

### Example Input Handling
```cpp
Command* handleInput()
{
  Unit* unit = getSelectedUnit();

  if (isPressed(BUTTON_UP)) {
    int destY = unit->y() - 1;
    return new MoveUnitCommand(unit, unit->x(), destY);
  }

  if (isPressed(BUTTON_DOWN)) {
    int destY = unit->y() + 1;
    return new MoveUnitCommand(unit, unit->x(), destY);
  }

  // Other moves...

  return NULL;
}
```

Commands being **one-use objects** is helpful when designing **undo systems**.

---

## Adding Undo Support

We extend the `Command` interface.
```cpp
class Command
{
public:
  virtual ~Command() {}
  virtual void execute() = 0;
  virtual void undo() = 0;
};
```

`undo()` reverses the state change made by the command.

---

### Move Command With Undo
```cpp
class MoveUnitCommand : public Command
{
public:
  MoveUnitCommand(Unit* unit, int x, int y)
  : unit_(unit),
    xBefore_(0),
    yBefore_(0),
    x_(x),
    y_(y)
  {}

  virtual void execute()
  {
    xBefore_ = unit_->x();
    yBefore_ = unit_->y();

    unit_->moveTo(x_, y_);
  }

  virtual void undo()
  {
    unit_->moveTo(xBefore_, yBefore_);
  }

private:
  Unit* unit_;
  int xBefore_, yBefore_;
  int x_, y_;
};
```

---

### Undo System

The game stores the **last executed command**.

When the user presses **Ctrl + Z**:
```
lastCommand->undo();
```

For **multiple undo levels**:

* Maintain a **list of commands**
* Undo → move backward in the list
* Redo → move forward
* Performing a new action **clears all future commands**

# Classy and Dysfunctional?

In languages with **closures**, the Command Pattern can be implemented using **functions instead of classes**.

### JavaScript Example
```js
function makeMoveUnitCommand(unit, x, y) {
  return function() {
    unit.moveTo(x, y);
  }
}
```

---

### JavaScript With Undo Support
```js
function makeMoveUnitCommand(unit, x, y) {
  var xBefore, yBefore;

  return {
    execute: function() {
      xBefore = unit.x();
      yBefore = unit.y();
      unit.moveTo(x, y);
    },

    undo: function() {
      unit.moveTo(xBefore, yBefore);
    }
  };
}
```
