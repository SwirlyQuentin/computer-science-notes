### **What is Software Architecture**

Less about writing code, more about **organizing it**.

#### **What is Good Architecture**

> "When you make a change, it's as if the entire program was crafted in anticipation of it."

- Tasks can be solved with just a few choice function calls that slot in perfectly
- Everything just fits together

*Architecture is about change.*

- Someone has to be modifying the codebase

**What makes good architecture is how easily it accommodates changes.**

#### **How do you make a change**

Before making a change, you have to understand what the existing code is doing.

- This is the hardest part
- Once you understand the problem *and the parts it touches*, the actual coding is sometimes trivial
- Usually when you make a change, there is a bit of reorganization to do in order to make your code fit with the rest of the program

#### **How can Decoupling Help**

When something is coupled, it's basically that you can't *fully* understand one without the other.

Decoupling allows for understanding and reasoning about either side *independently*.

The goal of software architecture:
**Minimize the amount of thinking and brain processing you need to do before you can make progress.**

Less coupling = less change that ripples.

### **At What Cost?**

Good architecture takes discipline.

- Every time you make a change or implement a feature, you have to work hard in order to integrate it with the rest of the program

> "It's not enough to just put in new plants. You must also weed and prune."

You have to think about:

- Which parts of a program should be decoupled
- How to introduce abstractions at those points
- Determine where extensibility should be engineered so future changes are easier to make

Adding a layer of abstraction = *speculating* that you will need that flexibility later.

- When that modularity doesn't end up being used, it can be hurtful:
  - More code to deal with
  - Layers of abstraction become what you think about

### **Performance and Speed**

Having many patterns hurt game performance?

Not necessarily

*The practice of optimization thrives on concrete limitations.*

- Knowing exactly what you need allows you to explicitly plan around it
  - Example: Having no more than 256 enemies means you can fit an ID into a single byte

The faster you can try out ideas, the faster you can "find the fun."


### **The Good in Bad Code**

Writing good architecture and maintaining it takes *time and effort*.

Game design requires a lot of trial and error, and you may not want to stay in a codebase for long.

Prototyping is a very useful programming practice. **However:**

- If you write throwaway code, you must be able to throw it away

### **Striking a Balance**

1. We want nice architecture so code is easier to understand over the lifetime of the project
2. We want fast runtime performance
3. We want to get today's features done quickly


These goals are somewhat in opposition.

- No simple answer, just tradeoffs

### **Simplicity**

Simplicity can ease these constraints:

- Try to write the cleanest, most direct solution to the problem
- The kind of code where, after you read it, you understand exactly what it does and can't imagine any other solution

Simple = less code overall.

Rarely are we presented with an elegant problem, only *different use cases* or:

- A long list of example behaviors

Don't:

- Just code the use cases one by one

Find the small bit of logic that can cover a large space of use cases.


### **Tips**

- Abstraction and decoupling make evolving your program faster and easier, but don't waste time on them unless you are confident the code needs the flexibility
- Think about and design for performance throughout development cycle, but put off low-level, nitty gritty optimizations that lock up your code until as late as possible
- Move quickly to explore your game's design space, but don't leave a mess behind
- If you are going to ditch code, don't make it pretty
- **If you make something fun, have fun making it**
